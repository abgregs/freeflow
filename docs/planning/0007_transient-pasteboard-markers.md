# Planning: Transient Pasteboard Markers (roadmap 0007)

Upgrade the pasteboard trade-off recorded in
[0006_runtime-security-hardening.md](0006_runtime-security-hardening.md):
dictations transit `NSPasteboard.general`, so clipboard-history managers
(Raycast, Alfred, Maccy, Paste) silently record everything the user dictates.
That collides with the privacy promise — the app keeps data on-device, but
another app may be logging it in ways the user doesn't expect.

## Mechanism

The [nspasteboard.org](http://nspasteboard.org/) convention defines marker
types that signal "don't record this" to clipboard managers:

- `org.nspasteboard.TransientType` — momentary content; history apps should
  not record it. Exactly our case: write → paste → restore within ~250 ms.
- `org.nspasteboard.ConcealedType` — confidential content (password-manager
  convention). Dictation can contain anything, so it applies.
- `org.nspasteboard.AutoGeneratedType` — programmatically generated content.

Declare all three alongside the string on **both** writes in
`TextInsertionManager`:

1. **The transcription write** — keeps dictations out of well-behaved
   clipboard histories.
2. **The snapshot restore** — prevents the restore from re-logging the user's
   original clipboard as a "new" copy. (The manager already recorded the
   original when the user first copied it.)

**Why this is a marking, not a guarantee:** the convention is voluntary.
Alfred (fixed in 3.4), Maccy, and Paste respect it; Raycast's support is
undocumented and must be tested empirically. A manager that ignores the
convention — or any process polling the pasteboard during the window — still
sees the text. The README caveat is *reworded*, not removed.

## Implementation sketch

- `TextInsertionManager` is the single pasteboard write site; the markers are
  added where the transcription string and the restored items are written.
- The restore must stay byte-faithful for the user's original content (RTF /
  images / custom UTIs round-trip unchanged) — markers are additional types,
  not replacements. Existing `TextInsertionManagerTests` must stay green.
- The marker-type list is a pure constant — unit-test that both write paths
  declare it.

## Empirical verification

1. With a marker-respecting manager (Maccy or Alfred): dictate; the
   transcription must not appear in history, and the restored original must
   not create a duplicate entry.
2. With Raycast: same test; document the observed behavior in this file. If
   Raycast ignores the markers, file an upstream issue — it affects every
   password manager user, not just Free Flow.
3. Paste behavior in target apps is unchanged (markers are invisible to the
   paste consumer).

## Acceptance criteria

1. Both pasteboard writes carry all three marker types; a unit test covers it.
2. Dictations stop appearing in at least one marker-respecting clipboard
   manager, verified on-device.
3. The README clipboard caveat is reworded to: marks writes as transient and
   concealed, which well-behaved clipboard managers respect; managers that
   ignore the convention may still record dictations.
4. The pasteboard trade-off entry in
   [0006_runtime-security-hardening.md](0006_runtime-security-hardening.md)
   points here instead of calling the exposure fully accepted.
