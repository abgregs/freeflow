## Status

**M9 (Tap modes + activation-mode picker) complete** (2026-06-09). `TapStateMachine` (pure, clock-injectable) interprets completed taps for **Single Tap** / **Double Tap**; `HotkeyManager` routes the tap modes through it while Hold stays inline. The **Activation Mode** picker ships in `SettingsView` with a mode-aware Caps Lock/Hold warning; `activationMode` persists via `SettingsStore`, which gained a small `DefaultsConvertible` protocol so `RawRepresentable` settings round-trip through `UserDefaults`. **`doubleTapWindowMs` is an internal `Constants` tunable, not a setting** — no slider, by decision (it's friction for a value the user shouldn't have to reason about). **Mid-recording reconfiguration is now live-or-deferred:** a key/mode change during a *tap*-mode recording applies **in place** — the watched key/mode is swapped while the one event tap keeps running (no rebuild, no dropped audio) — and the session posts a message on a new `notices` channel (`AppState.notice` → menu dropdown now, the recording HUD later) telling the user the newly-set key now stops the recording; **Hold** still defers until the held key is released. This also corrected a stale doc claim that a key change *rebuilds* the tap: it's always an in-place refilter, now consistent across [../architecture/threading-invariant.md](../architecture/threading-invariant.md), [../architecture/configuration.md](../architecture/configuration.md), and anti-pattern #7. Two post-review cleanups followed: the pure activation tables (`ActivationKeyOption`, `ActivationNotice`) moved to `Models/` so the session no longer reaches into `Views/` for notice copy, and deferred reconfigurations now **accumulate** (a key *and* mode change during one Hold recording both apply, not just the last). Verified: build + 92 tests green, plus on-device (all three modes, live key-change while recording, and the Caps Lock/Hold warning).

**M8 (Settings UI + custom dictionary) complete** (2026-06-08). `SettingsView` ships the Activation **key** picker (+ the verbatim Caps Lock/Hold warning from a pure, tested `ActivationKeyOption` table), the **Custom Dictionary** (list with a per-row delete button + add field), and **launch-at-login** (`SMAppService`). `@AppStorage(Settings.x.name)` binds the `Int`/`Bool` keys; the `[String]` dictionary binds through the typed `SettingsStore` via a small `@Observable` `DictionaryModel` (`@AppStorage` can't represent `[String]`). A latent bug surfaced first: `SettingsStore` never bridged external `@AppStorage` writes to its publishers (the `observation` field was declared, never wired) — fixed by observing `UserDefaults.didChangeNotification`, so a key change from the UI actually reaches `FreeFlowSession`. The custom dictionary then exposed three more: (1) `loadModel()` lacked `load: true`, so the model never loaded at launch and `buildPromptTokens` read a nil tokenizer → an empty, inert prompt; (2) `base.en` degenerates to empty output under a prompt, so the default moved to **`small.en`** (~240 MB) — the dictionary effectively requires it (see [../requirements/custom-dictionary.md](../requirements/custom-dictionary.md)); (3) `transcribe` now falls back to an unprompted decode when a prompted run is empty, so the dictionary can only help. A gated A/B eval harness (`DictionaryEvalTests`) verified biasing ("Veed" → "Vite" on `small.en`). **Deferred:** mode picker + `doubleTapWindowMs` (M9), the model picker (M9+), pause-media ([0003](0003_pause-media-while-dictating.md)), and the model-loading indicator the `load: true` change exposed ([0004](0004_model-loading-indicator.md)). Verified: build + 72 tests green; on-device settings + transcription confirmed.

**Menu-bar visual state complete** (2026-06-06). The full cycle now does visible work end-to-end. `AppState` (`@MainActor @Observable`) bridges `FreeFlowSession`'s Combine publishers (`state` + the new `errors`) to SwiftUI; the session imports no SwiftUI and stays the test surface. `MenuBarPresentation.visual(state:hasError:)` is a pure, unit-tested mapping to the `MenuBarExtra` icon + label — `mic` / `mic.fill` / `ellipsis`, "Ready" / "Recording..." / "Processing..." per [../requirements/core-feature.md](../requirements/core-feature.md) item 5 — with a warning glyph overriding only the idle icon. `FreeFlowSession.errors` emits a typed `FreeFlowError` (`.audioCapture` / `.transcription` / `.textInsertion`) per stage without blocking the return to `.idle` — the session-level error publisher M7 deferred. The message is path-redacted exactly once at the `AppState.apply(_:error:)` boundary before it reaches the menu ([ADR 0002](../decisions/0002-log-redaction-over-debug-flag.md)) and cleared when a new recording starts. See [../architecture/app-state-and-menu-bar.md](../architecture/app-state-and-menu-bar.md). Verified: build + 63 tests green, plus on-device end-to-end (menu-bar icon cycles `mic` → `mic.fill` → `ellipsis` with matching labels across a live recording cycle).

**M7 complete** (2026-06-04). Text insertion closes the cycle. `TextInsertionManager.insertText(_:)` is async: it snapshots the **full pasteboard** (every item, every type, as opaque `Data` so RTF / images / custom UTIs round-trip unchanged), writes the transcription, posts a synthesized ⌘V down/up through the capability, sleeps `Constants.clipboardRestoreDelay` (250 ms) so the target app's paste handler can pull the new contents in, then restores the snapshot. **Restore-on-throw:** any failure before the restore (a post error, or sleep cancellation) still restores the original pasteboard in the `catch` before rethrowing — stranding the transcription in the user's clipboard is a worse failure than the paste error itself. `AccessibilityCapability.postKeyEvent(...)` is now the **sole `CGEvent.post` call site** (load-bearing rule #3). It guards on `.granted`, then runs the **silent-no-op probe**: synthesizes a Shift modifier-down via `.cghidEventTap`, reads `CGEventSource.flagsState(.combinedSessionState)`, and refuses the post (downgrading `status` to `.denied`, throwing `.silentNoOp`) if the OS never reflected the synthesized modifier — the structural detector for anti-pattern #3 (a bundle that signs and trusts but silently no-ops). The result is cached in `probeConfirmed`: it runs once at the first `postKeyEvent` after `.granted`, then memoizes, and resets whenever `recheck()` sees the OS view drop below `.granted`. A CapsLock toggle pair is the documented plan-B if the Shift technique proves unreliable. `OnboardingCoordinator` is **extracted from `AppDelegate`** (closes the planning TODO): it owns the onboarding gate, the `NSWindow` lifetime, and a runtime subscription to every capability's `status` publisher that **re-presents onboarding on a `.granted → !.granted` transition** — the user-facing surface for the probe firing mid-session. `AppDelegate` is back to a thin lifecycle shell. M7 shipped a `DICTATION_VERBOSE_LOGS` build flag to gate `privacy: .public` on error strings; it was **superseded immediately after** ([ADR 0002](../decisions/0002-log-redaction-over-debug-flag.md)) by source-level path redaction. Error strings now log `privacy: .public` everywhere but pass through `LogRedaction.redactUserPaths(_:)`, which strips `/Users/<name>` paths — so failure reasons stay readable in field bug reports without leaking the account name, and user content stays protected by *omission* (only `.count` is ever logged) per [../conventions/logging.md](../conventions/logging.md). The pre-M3 `appState.errorMessage` error-surface sketch was reconciled in the pipeline doc as a **deliberate deferral, not a regression**: transient post failures were log-only at M7; the session-level error publisher landed with the menu-bar visual-state milestone (above). Verified: build + 52 tests green, plus on-device end-to-end (real Accessibility grant → hold-speak-release → transcribed text pasted at the cursor, original clipboard restored).

**M6 complete** (2026-06-03). WhisperKit 0.18.0 integrated. `TranscriptionService` exposes explicit `loadModel()` (idempotent, coalesces concurrent callers) and `transcribe(audioSamples:) -> String`; `customDictionaryTerms` is a property + setter ready for M8 to wire `Settings.customDictionaryTerms`. The load-bearing **`filterSpecialTokens(_:specialTokenBegin:)`** static helper drops timestamp / language / sentinel tokens before they corrupt `DecodingOptions.promptTokens`. `AppDelegate` fire-and-forgets `loadModel` at launch on its own Task — first run downloads ~140 MB for `base.en` to WhisperKit's cache (`~/Documents/huggingface/models/argmaxinc/whisperkit-coreml/`). `FreeFlowSession.handleDeactivate` calls `transcribe` after `stopRecording` in a nested catch — audio and transcription failures are independent; either path still returns to `.idle` and applies pending reconfig. Typed errors: `TranscriptionError.modelNotLoaded` / `.transcriptionFailed(underlying:)` / `.emptyTranscription`. M6 deliberately keeps the explicit lifecycle (no `skipModelLoadForTesting` flag) — see [ADR 0001](../decisions/0001-defer-cycle-protocol-seams.md). Verified: build + 47 tests green, plus on-device end-to-end (Right Option → recording → transcribed-char log in `log stream`).

**M5 complete** (2026-06-02). `MicrophoneCapability` owns `AVAudioEngine` start/stop and exposes an `audioBuffers` publisher. `AudioCaptureManager` subscribes during `.recording`, waits up to 300 ms for the first buffer (engine-warmup race), then converts hardware-format buffers to 16 kHz mono Float32 via `AVAudioConverter`. `FreeFlowSession.handleDeactivate` is async: `.recording → .processing → .idle`. The M3 deferral loop closes here too — `pendingReconfiguration` fires on every return to `.idle`.

**M4 complete** (2026-06-01). `InputMonitoringCapability` owns the system-wide `CGEventTap` on a dedicated `com.freeflow.eventtap` background thread; the C tap callback hops to the main actor before publishing `TapEvent`s. `HotkeyManager` interprets `.flagsChanged` for the watched keycode (Hold mode only — tap modes in M9) and fires semantic `onActivate`/`onDeactivate`. ## Notes for the agent picking this up

- The architecture docs encode hard-won lessons from a predecessor implementation. Treat the **Why:** annotations seriously — they exist because someone got bitten.
- The most common failure mode in this app's lineage is infrastructure that *looks* like it works but silently doesn't (a tap that fires but a paste that's blocked; a bundle that signs but has no identifier). Verify with `codesign -dv`, log inspection, and end-to-end tests in a real text field — not just "swift build succeeded."
- If you find a gap in the docs while working, flag it in `/debrief` rather than just plowing through. The docs are designed to evolve with the code.
